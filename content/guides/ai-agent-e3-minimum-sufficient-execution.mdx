[LLM 에이전트](/glossary/ai-agent)가 다단계 엔지니어링 작업을 대신 처리하는 일이 늘고 있지만, 정작 "이 작업이 얼마나 어려운가"를 스스로 판단하는 능력은 거의 없다는 지적이 나왔다. 대부분의 에이전트는 이미 읽은 파일과 의존성을 다시 훑는 최대-맥락-우선 전략을 쓰며, 한 줄짜리 수정 요청조차 코드베이스 전반을 감사하는 작업으로 부풀린다.

이 연구는 이 문제를 작업 인지 실행 범위 추정(task-aware execution-scope estimation)이라는 개념으로 정식화하고, 최소 충분 실행(minimum-sufficient execution)이라는 목표를 세운다. 에이전트 활용이 실무 워크플로 전반으로 확산되는 지금, 불필요한 과잉 탐색은 비용과 지연 시간으로 고스란히 쌓이기 때문에 중요한 문제다.

| 항목 | 수치 |
| --- | --- |
| 테스트 규모 | 결정론적 편집 121건 (MSE-Bench) |
| 성공률 | 최강 베이스라인과 동률 100% |
| 비용 절감 | 85% |
| 토큰 절감 | 91% |

## 1. 무엇이 문제인가 — 에이전트의 인지적 과잉

연구진은 에이전트가 겪는 이 낭비를 정량화하기 위해 에이전트 인지 중복 비율(Agent Cognitive Redundancy Ratio, ACRR)이라는 지표를 제안한다. 작업이 실제로 필요로 하는 정보량과, 에이전트가 실제로 훑는 정보량 사이의 격차를 재는 척도다.

문제의 핵심은 에이전트가 커밋 전에 "이 작업의 최단 신뢰 경로가 무엇인가"를 먼저 묻지 않는다는 데 있다. 그 결과 난이도와 무관하게 항상 최대 범위의 맥락을 먼저 확보하는 습관이 굳어진다.

## 2. 얼마나 벌어지나 (비교) — E3 vs 베이스라인

MSE-Bench는 역량이 통제된 시뮬레이터 안에서 121건의 편집 작업으로 구성된 결정론적 벤치마크다. 여기서 E3는 가장 강한 베이스라인과 동일한 100% 성공률을 유지하면서도 비용·토큰·검토 파일 수를 크게 줄였다.

| 지표 | E3의 결과 |
| --- | --- |
| 성공률 | 최강 베이스라인과 동일(100%) |
| 비용 | 85% 감소 |
| 토큰 사용량 | 91% 감소 |
| 검토한 파일 수 | 92% 감소 |
| 적응형 검색 베이스라인 대비 | 16% 우위 |

> **요약**: E3는 성공률을 떨어뜨리지 않으면서 탐색 비용만 대폭 줄였고, 이 우위는 지시문 표현을 바꾸거나 비용 가중치를 달리해도 거의 그대로 유지됐다.

## 3. 어떻게 작동하나 — Estimate, Execute, Expand

E3는 이름 그대로 세 단계로 움직인다. 먼저 초기 작업 지점을 추정(Estimate)하고, 최소한으로 실행 가능한 경로를 실행(Execute)한 뒤, 검증에 실패했을 때만 범위를 확장(Expand)한다. 즉 처음부터 모든 것을 훑는 대신 "일단 최소한으로 해보고, 안 되면 넓힌다"는 순서를 따른다.

이 구조는 작업을 과대평가해 낭비하는 것도, 과소평가해 실패하는 것도 막도록 설계됐다. 검증 실패라는 신호가 있을 때만 추가 탐색 비용을 지불하기 때문에, 쉬운 작업에서는 비용이 크게 줄고 어려운 작업에서는 여전히 확장 경로가 열려 있다.

> **주의**: 이 수치는 역량이 통제된 시뮬레이터 벤치마크에서 나온 결과이며, 저자들도 실제 배포된 에이전트의 측정치가 아니라 실행 중복(execution redundancy)에 대한 통제된 검증이라고 명시한다.

## 4. 실무 의미 — 실제 모델로도 재현됐나

연구진은 시뮬레이터 결과를 검증하기 위해 실제 오픈소스 라이브러리를 편집하는 gpt-4o 에이전트로 별도의 실측 하네스(LLM-Case)를 구성했다. 여기서는 후보 패치마다 해당 프로젝트의 실제 pytest 스위트를 돌려 측정된 오라클과 대조해 채점했다.

실제 모델 환경에서는 과잉 탐색 정도가 시뮬레이터보다는 완만했지만 여전히 실재했고, E3는 비교 가능한 성공률에서 가장 가볍고 빠른 정책이었다. E3의 유일한 실패 사례도 잘못된 편집이 아니라 API 제공자의 속도 제한(rate-limit) 때문이었다는 점이 눈에 띈다.

## 마무리 — agenwiki의 한줄평

에이전트에게 필요한 것은 더 큰 맥락 창이 아니라 "이 작업이 얼마나 쉬운지 먼저 가늠하는 판단력"이라는 점을 수치로 보여준 연구다. 비용·토큰 절감폭이 크다는 점에서, 에이전트 파이프라인을 운영하는 팀이라면 실행 전 난이도 추정 단계를 눈여겨볼 만하다.

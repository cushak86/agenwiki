보안 [에이전트](/glossary/ai-agent) 평가는 보통 추론 예산을 넉넉히 준 상태에서 최고 성능만 측정해왔다. 취약점 발견, 익스플로잇 개발, 침투 테스트, CTF 완료 같은 최상의 결과를 기준으로 삼는 방식이다. 이 연구는 그 방식이 유용하지만 불완전하다고 지적한다. 실제 보안 운영에서는 추론 한 스텝, 도구 호출 한 번, 텔레메트리 조회 한 건마다 예산이 소모되기 때문이다.

연구진은 공격형 Cybench 챌린지와 방어형 Splunk BOTS v1 조사 챌린지를 비용-성공 관점에서 재평가했다. 최고 성공률 한 줄만 보고하는 대신, 모델들을 동일한 비용 수준에서 비교하고 성능을 추론 지출과 도구 지출로 나눠 분해했다.

| 항목 | 수치 |
| --- | --- |
| 평가 대상 | 공격(Cybench) · 방어(Splunk BOTS v1) |
| 비교 기준 | 고정 비용 수준별 성공률 |
| 분해 축 | 추론 지출 vs 도구 지출 |
| 핵심 발견 | 레드팀·블루팀 과제의 스케일링 양상이 다름 |

## 1. 무엇이 문제인가 — 성공률만 보는 평가의 한계

기존 보안 에이전트 벤치마크는 넉넉한 추론 예산 아래서의 최고 성능, 즉 정점 공격 역량을 재는 데 초점을 맞춰왔다. 하지만 실제 보안 운영에서는 매 추론 단계와 도구 호출, 텔레메트리 조회, 정보 보강 요청마다 예산이 든다. 최고 성능만 보는 지표는 이 운영 비용을 반영하지 못한다는 것이 연구진의 문제의식이다.

## 2. 얼마나 벌어지나 (비교)

| 공격(레드팀) | 방어(블루팀) |
| --- | --- |
| 테스트 타임 연산을 늘릴수록 성능 향상 | 추론 예산만으로는 성능이 같은 방식으로 늘지 않음 |
| 확장된 오픈웨이트 모델이 프론티어 독점 모델에 근접 | 절제된 도구 사용·텔레메트리 탐색·선택적 보강이 더 중요 |
| 비용 경쟁력을 유지하며 성능 확보 가능 | 원시 추론 예산만으로는 부족 |

> **요약**: 공격형 과제는 돈을 더 쓸수록 성능이 오르지만, 방어형 과제는 그렇지 않다는 것이 이번 비교의 핵심 발견이다.

## 3. 어떻게 작동하나 — 비용 수준을 고정한 비교

연구진은 Cybench의 공격 CTF 챌린지와 Splunk BOTS v1의 방어 SOC 조사 챌린지를 대상으로, 여러 모델을 동일한 비용 수준에 맞춰 비교했다. 성능을 단일 지표가 아니라 추론 지출과 도구 지출로 나눠 살핀 것이 방법의 핵심이다. 그 결과 공격 과제에서는 테스트 타임 연산을 늘리는 만큼 성능이 따라오고, 확장된 오픈웨이트 모델도 비용 경쟁력을 유지하며 프론티어 독점 모델에 근접할 수 있었다. 반면 방어 SOC 조사 과제에서는 성공이 원시 추론 예산보다 절제된 도구 사용, 텔레메트리 탐색, 선택적 정보 보강에 더 크게 좌우됐다.

> **주의**: 방어 과제에서 추론 예산을 늘리는 것만으로는 공격 과제와 같은 폭의 성능 향상을 기대하기 어렵다는 점이 이번 결과의 시사점이다.

## 4. 실무 의미 — 경제성과 운영 적합성을 함께 봐야 한다

연구진은 보안 에이전트 벤치마크가 과제 성공률뿐 아니라 경제적 효율성과 운영 적합성도 함께 측정해야 한다고 주장한다. 비용을 고려한 SOC 환경 맞춤 평가라야 어떤 모델이 지금 실제로 쓸 만한지, 방어형 에이전트가 어디서 더 개선돼야 하는지를 더 분명하게 보여준다는 것이다. 연구진은 결과를 담은 인터랙티브 웹사이트도 함께 공개했다.

> **팁**: 보안 에이전트를 도입·평가하는 팀이라면 최고 성공률 수치만 볼 게 아니라, 자체 운영 예산 수준에서의 비용 대비 성능을 함께 확인할 필요가 있다.

## 마무리 — agenwiki의 한줄평

같은 "AI 보안 에이전트"라도 공격은 돈을 태울수록 강해지고 방어는 그렇지 않다는 비대칭은, 방어 자동화를 준비하는 조직이 특히 새겨들을 만한 결과다. 성공률 한 줄에 기대지 말고 비용 구조까지 함께 보라는 이 논문의 제안은 보안 에이전트 평가 관행에 실질적인 수정을 요구한다.

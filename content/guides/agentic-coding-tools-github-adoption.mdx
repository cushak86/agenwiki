[AI 에이전트](/guides/what-is-llm)가 풀 리퀘스트(PR)를 직접 만들어 제출하는 사례가 늘고 있다는 인식은 이제 낯설지 않다. 그러나 이런 에이전틱 코딩 도구가 실제로 깃허브 생태계 전반에서 얼마나 널리 쓰이는지는 따로 확인된 적이 드물었다. 이 논문은 인기 저장소 2,361곳에서 발생한 에이전트 생성 PR 25,264건을 3개월 구간으로 추적해 이 공백을 메운다.

결과는 "빠른 확산"이라는 통념과 다소 거리가 있다. 채택은 소수 프로젝트에 집중돼 있고, 규모가 작은 팀일수록 오히려 더 적극적으로 도구를 쓰는 경향이 나타났다. 이 글에서는 채택 규모, 프로젝트 규모별 차이, 그리고 사람과 에이전트가 실제로 어떻게 협업하는지를 차례로 살펴본다.

## 1. 무엇을 조사했나

연구진은 인기 깃허브 저장소 2,361곳에서 발생한 에이전트 생성 PR 25,264건을 수집해 세 가지를 들여다봤다. 에이전틱 코딩 도구의 채택 양상, 프로젝트 단위의 PR 생산성, 그리고 사람과 에이전트 사이의 협업 패턴이다. 단순히 "에이전트가 PR을 몇 개 만들었는가"를 세는 데 그치지 않고, 그 PR들이 프로젝트 안에서 어떻게 관리되고 누가 검토하는지까지 함께 분석했다는 점이 이 연구의 핵심이다.

## 2. 얼마나 쓰이고 있나 (비교)

3개월 관측 기간 동안 중간값 저장소는 에이전트 PR을 단 1~2건만 만들었다. 즉 절반이 넘는 저장소에서 에이전틱 코딩 도구의 사용이 아직 실험적 수준에 머물러 있다는 뜻이다. 반면 업계에서 보고된 추정치인 "참여자당 3개월 PR 36건"을 넘어서는 프로젝트는 소수에 그쳤고, 대다수 프로젝트는 이 기준선 아래에 머물렀다.

프로젝트 규모별 차이도 뚜렷했다.

| 구분 | 에이전트 PR 참여·활동 수준 |
| --- | --- |
| 소규모 프로젝트 (기여자 1~5명) | 참여 비율·평균 활동량 모두 높음 |
| 중형·대형 프로젝트 | 참여 비율·평균 활동량 낮음 |

소규모 팀에서 참여 비율과 평균 활동량이 모두 더 높다는 것은, 의사결정 구조가 단순한 팀일수록 새 도구를 실험적으로 들여오는 문턱이 낮다는 뜻으로 읽을 수 있다.

> **요약**: 인기 저장소 2,361곳을 봤을 때, 중간값 저장소는 3개월간 에이전트 PR을 1~2건만 만들었고 소규모 팀이 오히려 더 적극적이었다.

## 3. 사람과 에이전트는 어떻게 일하나

협업 패턴에서 가장 두드러진 특징은 "단일 인간 감독" 모델이 지배적이라는 점이다. 개발자 한 명이 에이전트의 결과물을 검토하고 필요하면 직접 수정하는 방식이 대부분을 차지했고, 여러 사람이 함께 에이전트의 작업을 검토·조율하는 다인 협업 패턴은 흔치 않았다. 이는 에이전트가 만든 코드를 조직 차원에서 분산 검증하는 체계가 아직 자리 잡지 않았다는 신호로 볼 수 있다.

## 4. 실무 의미

이 결과가 실무자에게 던지는 메시지는 두 가지다. 첫째, 에이전틱 코딩 도구의 실제 확산 속도는 도구 자체의 성능만으로 결정되지 않는다. 저장소 대부분이 아직 기준선 아래에 머무는 이유는 조직 프로세스와 검토 체계가 도구 발전 속도를 따라가지 못하고 있기 때문일 가능성이 크다. 둘째, 지금 구조에서는 검토자 한 명이 사실상 모든 병목을 떠안고 있다는 점을 감안해야 한다. 팀 규모가 커질수록 이 단일 감독 모델이 그대로 유지될 수 있을지는 별도로 검증이 필요하다.

> **주의**: 이 수치는 2026년 초 시점의 스냅숏이다. 저자들도 이는 초기 관측일 뿐이며, 채택 패턴이 시간에 따라 달라질 수 있다고 명시하고 있다.

## 마무리 — agenwiki의 한줄평

에이전틱 코딩 도구를 둘러싼 논의는 종종 "얼마나 잘 작동하는가"에 집중되지만, 이 연구가 보여주는 병목은 오히려 조직과 인간 검토 체계 쪽에 있다. 도구 도입을 검토하는 팀이라면 에이전트의 코딩 능력보다 "누가, 어떻게 결과물을 검증할 것인가"를 먼저 설계하는 편이 현실적이다.

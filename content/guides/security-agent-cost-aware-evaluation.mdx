지금까지 보안 에이전트 평가는 대부분 넉넉한 추론 예산 아래서 최고 성능만 측정해왔다. 취약점 발견, 익스플로잇 개발, 침투 테스트, CTF 완주처럼 최상의 공격 능력을 보여주는 지표들이다. 연구진은 이 방식이 유용하지만 불완전하다고 지적한다. 실제 운영 환경의 보안 업무에서는 추론 한 단계, 도구 호출 한 번, 텔레메트리 조회 한 건, 정보 보강 요청 한 건까지 전부 예산을 소모하기 때문이다.

이 연구는 언어모델 기반 보안 에이전트를 '비용-성공' 관점에서 다시 평가했다. 공격 쪽은 Cybench 침투 테스트 과제, 방어 쪽은 Splunk BOTS v1 SOC 조사 과제를 무대로 삼았다. 최고 성능 한 지점만 보고하는 대신, 동일한 비용 수준에서 모델들을 비교하고 성능을 추론 지출과 도구 지출로 나눠 분해했다.

그 결과 공격(레드팀)과 방어(블루팀) 과제가 서로 다른 방식으로 확장된다는 사실이 드러났다. 이 차이는 보안 에이전트를 실제로 도입하려는 조직에 직접적인 함의를 준다.

## 1. 무엇을 다시 측정했나

기존 보안 에이전트 벤치마크는 "얼마나 잘하는가"만 보고 "얼마나 들여서 잘하는가"는 보지 않았다. 연구진은 언어모델 보안 에이전트를 Cybench(공격)와 Splunk BOTS v1(방어) 두 무대에서 실행하면서, 성공 여부와 함께 추론 비용·도구 비용을 동시에 기록했다. 최상위 한 번의 성공이 아니라, 고정된 비용 수준마다 어떤 모델이 얼마나 성공하는지를 비교한 것이 핵심 방법론이다.

## 2. 얼마나 벌어지나 (비교)

공격과 방어 과제는 비용을 늘렸을 때 반응하는 방식이 근본적으로 달랐다.

| 구분 | 공격(레드팀) — Cybench | 방어(블루팀) — Splunk BOTS v1 |
| --- | --- | --- |
| 성능 확장 방식 | 테스트 타임 연산량을 늘릴수록 성공률 상승 | 연산량을 늘려도 같은 방식으로 확장되지 않음 |
| 성공을 좌우하는 요인 | 추가 추론 예산 | 절제된 도구 사용, 텔레메트리 탐색, 선별적 정보 보강 |
| 오픈웨이트 모델의 위치 | 스케일을 키우면 프런티어 독점 모델에 근접하면서도 비용 경쟁력 유지 | 해당 경향 확인되지 않음 |

즉 공격 과제는 "돈을 더 쓰면 더 잘한다"는 익숙한 스케일링 법칙을 따랐지만, 방어 과제인 SOC 조사는 원시 추론 예산보다 도구를 얼마나 절제되고 정확하게 쓰는지가 성공을 갈랐다.

> **요약**: 공격형 보안 에이전트는 연산을 더 쓸수록 강해지지만, 방어형(SOC 조사) 에이전트는 연산량이 아니라 도구를 얼마나 절제되게 쓰는지가 성공을 결정한다.

## 3. 어떻게 측정했나

연구진은 성공률 단일 지표 대신 비용-성공 곡선을 사용했다. 같은 비용 수준에서 여러 모델의 성공률을 나란히 비교하고, 그 비용을 다시 추론 지출(모델이 얼마나 오래·많이 사고하는가)과 도구 지출(외부 도구·텔레메트리 조회를 얼마나 쓰는가)로 쪼갰다. 이렇게 분해하면 어떤 모델이 "생각을 많이 해서" 잘하는지, "도구를 잘 골라 써서" 잘하는지가 구분된다.

> **주의**: 이 연구는 Cybench와 Splunk BOTS v1이라는 특정 벤치마크 두 개에서 관찰된 경향이다. 모든 공격·방어 시나리오에 동일한 비용-성능 관계가 성립한다고 논문이 일반화하지는 않는다.

## 4. 실무 의미

보안팀이 에이전트 도입을 검토할 때, 리더보드 최고 점수만 보고 결정하면 실제 운영 비용을 과소평가할 위험이 있다. 연구진은 보안 에이전트 벤치마크가 과제 성공률과 함께 경제적 효율성, 운영 환경과의 적합성을 함께 측정해야 한다고 주장한다. 특히 방어(SOC) 업무에서는 연산 예산을 키우는 것보다 도구 사용을 얼마나 정교하게 설계하는지가 실제 성능을 좌우한다는 점을, 도입 전에 확인할 필요가 있다.

## 마무리 — agenwiki의 한줄평

"더 많이 생각하면 더 잘한다"는 공식이 공격 에이전트에는 통하지만 방어 에이전트에는 통하지 않는다는 이 구분은, 보안 AI를 저울질하는 조직이 벤치마크 1등이 아니라 비용 대비 실제 업무 적합성을 기준으로 삼아야 한다는 실용적 경고로 읽힌다.

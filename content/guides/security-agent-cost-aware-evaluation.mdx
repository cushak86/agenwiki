보안 분야의 언어모델 에이전트 평가는 지금까지 대개 "넉넉한 추론 예산 아래 최고 성능"을 기준으로 삼아 왔다 — 취약점 발견, 익스플로잇 개발, 침투 테스트, CTF 완주 같은 항목이 그것이다. 하지만 실제 보안 운영에서는 추론 한 스텝, 도구 호출 한 번, 텔레메트리 조회 한 번마다 비용이 든다. 이 연구는 "최고 성능"이 아니라 "비용 대비 성공"이라는 렌즈로 공격형·방어형 에이전트를 나란히 측정한다.

## 1. 무엇을 측정했나 — 비용-성공 렌즈

연구팀은 공격 측 과제로 Cybench(공격형 CTF 챌린지)를, 방어 측 과제로 Splunk BOTS v1(SOC 조사 챌린지)을 사용했다. 두 경우 모두 최고 성공률 하나만 보고하는 대신, 동일한 비용 수준에서 모델들을 비교하고 성능을 추론 지출과 도구 지출로 나눠 분해했다.

## 2. 얼마나 다른가 — 공격과 방어의 스케일링 비교

가장 눈에 띄는 결과는 레드팀과 블루팀 과제가 비용에 따라 전혀 다르게 반응한다는 점이다.

| 구분 | 비용을 늘렸을 때 |
| --- | --- |
| 공격(Cybench CTF) | 테스트타임 연산을 늘릴수록 성능 향상, 오픈웨이트 모델도 스케일링 시 프론티어 독점 모델에 근접 |
| 방어(Splunk BOTS SOC 조사) | 같은 방식으로 스케일링되지 않음 — 원시 추론 예산보다 도구 사용의 규율이 성패를 가름 |

즉 공격 쪽은 "돈(연산)을 더 쓰면 더 잘한다"는 익숙한 스케일링 법칙을 따르지만, 방어 쪽은 텔레메트리 탐색과 선택적 조사(enrichment)를 얼마나 절제 있게 하느냐가 원시 추론량보다 더 중요하게 작용한다.

> **요약**: 공격형 에이전트는 비용을 늘리면 성능이 오르지만, 방어형 SOC 조사 에이전트는 도구 사용의 규율이 관건이라 같은 방식으로 개선되지 않는다.

## 3. 왜 이런 차이가 나나 — 도구 사용의 규율 vs 추론 예산

CTF형 공격 과제는 익스플로잇 탐색·검증처럼 추론을 반복해서 밀어붙이면 성과가 쌓이는 구조에 가깝다. 반면 SOC 조사는 방대한 로그와 텔레메트리 중 어디를 볼지 골라내는 판단, 즉 "무엇을 조회하지 않을지"를 결정하는 절제가 성패를 좌우한다. 추론을 더 많이 투입한다고 이 판단력이 자동으로 좋아지지는 않는다는 것이 연구가 관찰한 비대칭의 핵심이다.

## 4. 실무 의미 — 보안 에이전트를 고를 때 볼 것

연구팀은 보안 에이전트 벤치마크가 과제 성공률뿐 아니라 경제적 효율성과 운영 적합성을 함께 측정해야 한다고 주장한다. 비용을 고려한 SOC 중심 평가라야 어떤 모델이 지금 실제로 쓸 만한지, 방어형 에이전트가 어디서 더 개선돼야 하는지가 분명해진다는 것이다.

> **팁**: 방어형(블루팀) 에이전트를 도입할 때는 최고 성공률 숫자보다 "같은 비용에서 도구 호출을 얼마나 절제 있게 쓰는가"를 함께 살펴보는 것이 실무적으로 더 의미 있는 기준이 될 수 있다.

## 마무리 — agenwiki의 한줄평

"더 많이 쓰면 더 잘한다"는 스케일링 직관이 공격에는 통하고 방어에는 통하지 않는다는 비대칭은, 보안 [AI 에이전트](/glossary/ai-agent) 도입을 검토하는 조직이 성공률 하나만으로 모델을 고르면 안 되는 이유를 잘 보여준다.
